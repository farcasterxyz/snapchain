# L0 conformance corpus

A versioned, checked-in set of golden vectors that pin snapchain's byte-exact output for a fixed set
of inputs. It is the executable embodiment of the Farcaster protocol spec
([`farcasterxyz/protocol/docs/SPECIFICATION.md`](https://github.com/farcasterxyz/protocol/blob/main/docs/SPECIFICATION.md)),
and the artifact an alternative client runs in its own CI to prove parity (FIP validator-onboarding,
gate **L0** — NEYN-12044 / [snapchain#917](https://github.com/farcasterxyz/snapchain/issues/917)).

The canonical home for these vectors is the `vectors/` directory of `farcasterxyz/protocol`; the copy
here is vendored so snapchain can golden-test offline. A CI drift check keeps the two in lockstep.

## Layout

```
corpus/v<N>/
  manifest.json          # vector metadata + expected values (hex)
  messages/<id>.pb       # the encoded protobuf Message for each vector (authoritative bytes)
  messages/<id>.pb.txt   # human-reviewable wire disassembly of the .pb (review aid)
```

Each `<id>.pb` is the byte-exact wire `Message`; `<id>.pb.txt` is a deterministic, zero-dependency
disassembly committed beside it so corpus changes are auditable in a diff (a raw binary blob is
invisible in review). The `.pb` bytes and the `manifest.json` hex are the authoritative artifacts;
the `.pb.txt` is regenerated by the generator and verified against the renderer in CI.

## Two layers of conformance

**Crypto layer — encoding / hash / signature (cross-client, protocol-canonical).** Rests only on
protobuf + BLAKE3 + Ed25519, all of which have conformant implementations in every language. Any
client can verify this layer. See `client_parity_tests/conformance.test.ts` for an independent
(TypeScript / `@farcaster/core`) verifier.

**State-root layer — post-state merkle root (snapchain-internal, for now).** The shard trie root after
registering the fid and merging the message. snapchain's sharded trie key layout
(`[virtual_shard][fid][msg_type<<3][hash]`) diverges from the timestamp-prefixed Sync ID described
in protocol §4.2.1, which is stale. Until that spec is reconciled (tracked on epic NEYN-12026), the
roots here are reference-implementation-authoritative and verified by snapchain's Rust cross-node
test, not promoted to the protocol corpus. Vectors whose message needs extra on-chain preconditions
to merge (`key_add` / `key_remove`, and a `verification_add` with a real claim signature) ship
crypto-layer vectors only and omit `expected_state`.

## Manifest schema

```jsonc
{
  "corpus_version": 1,
  "protocol": { "network": "Devnet", "engine_version": "V18" },
  "vectors": [{
    "id": "cast_add_basic",
    "description": "...",
    "message_type": "CastAdd",
    "fid": 1234,
    "timestamp": 94608000,                 // fixed Farcaster-epoch seconds
    "signer_private_key": "<hex32>",        // Ed25519 secret key the message is signed with
    "input": { /* semantic body fields, for non-Rust clients reconstructing the message */ },
    "message_file": "messages/cast_add_basic.pb",
    "expected": {
      "data_bytes":    "<hex>",  // canonical MessageData serialization (the signing pre-image)
      "hash":          "<hex>",  // BLAKE3-160 of data_bytes
      "signature":     "<hex>",  // Ed25519 signature over hash
      "signer":        "<hex>",  // Ed25519 public key
      "message_bytes": "<hex>"   // full encoded Message (identical to message_file)
    },
    "expected_state": {          // omitted for crypto-layer-only vectors
      "shard_id": 1,
      "post_state_root": "<hex>"
    }
  }]
}
```

## Crypto-layer verification algorithm (language-agnostic)

Protobuf serialization is **not** canonical across implementations (e.g. prost and ts-proto differ
on empty `repeated` fields), so a client cannot reproduce snapchain's encoding from the semantic
fields. The Farcaster spec (§2 Hashing) accommodates this with the `data_bytes` field: the hash and
signature are computed over a carried serialization. Every corpus message therefore carries
`data_bytes` = snapchain's (prost) `MessageData` encoding. A conforming client:

1. Decodes the wire `Message` from `message_file` (proves wire-format interop).
2. Reads the carried `message.data_bytes` and checks it equals `expected.data_bytes`.
3. Computes `hash = BLAKE3(data_bytes)[0..20]` and checks it equals `expected.hash` and
   `message.hash`.
4. Verifies the Ed25519 `signature` over `hash` with `signer`, and checks the bytes equal
   `expected.signature` / `expected.signer`.
5. Decodes `data_bytes` back into a `MessageData` and checks the semantic fields (`fid`,
   `timestamp`, `type`, body) match.

A client must **not** assert that re-encoding `MessageData` reproduces `data_bytes` — that is
impossible across encoders and is the entire reason `data_bytes` exists.

## Regenerating

The Rust reference implementation is the source of truth. After an intentional protocol change:

```sh
cargo test --test conformance_test generate_corpus -- --ignored --nocapture
```

Then re-run the verifiers (`cargo test --test conformance_test`) and the TS verifier
(`cd tests/client_parity_tests && yarn install && yarn test`), and sync the corpus into
`farcasterxyz/protocol/vectors/`. The generator is deterministic: regenerating with no protocol
change produces a byte-identical corpus (CI enforces this).
